# カスタムハイパーリンク集 構築計画

## 1. プロジェクト概要

GoogleスプレッドシートでURLを一元管理し、GitHub Pages上にカード型のリンク集として表示する。

主な利用者は本人のみとし、複数のPC・スマートフォン・タブレットから、同じリンク環境へアクセスできる状態を構築する。

GitHub Pagesはインターネット上に公開されるが、積極的な一般公開は行わず、URLを知る本人が使用する前提とする。

---

## 2. 目的

- よく使用するURLへ最短でアクセスする
- 端末ごとにブックマークを再設定する作業をなくす
- URLを用途別に整理する
- 一時的に必要なURLをピン留めする
- タスク終了後にピン留めや表示を解除する
- GoogleスプレッドシートだけでURLを管理できるようにする
- 表示速度と保守性を重視し、機能を必要最小限にする

---

## 3. 基本構成

```text
Googleスプレッドシート
        ↓
Google Apps Script
        ↓ JSONデータ
GitHub Pages
        ↓
PC・スマートフォン・タブレット
```

### 各構成要素の役割

#### Googleスプレッドシート

URL、カテゴリ、表示名、ピン留め状態などを管理する。

#### Google Apps Script

スプレッドシートのデータを読み込み、GitHub Pagesから取得できるJSON形式で出力する。

#### GitHub Pages

JSONデータを取得し、カード型のWebページとして表示する。

---

## 4. 利用方針

### 想定利用者

- 原則として本人のみ
- 本人が所有または利用する複数端末
- 必要に応じてURLを知る限定的な相手

### 公開範囲

GitHub Pages自体には正式なアクセス制限を設けない。

ただし、以下の方法で実質的な限定利用とする。

- 推測されにくいリポジトリ名またはページURLを使用する
- 検索エンジンへの登録を抑制する
- 外部サイトからリンクしない
- SNSや公開資料にURLを掲載しない

この方式は非公開化ではない。URLが第三者へ渡った場合、その第三者も閲覧できる。

---

## 5. 画面構成

### 5.1 ヘッダー

ページ上部には以下のみを表示する。

- ページ名
- 最終データ更新日時
- 必要に応じて再読み込みボタン

検索欄や複雑なメニューは設けない。

### 5.2 ピン留め領域

現在進行中の作業や、一時的によく使用するURLをページ最上部に表示する。

表示順はスプレッドシートの並び順で管理する。

### 5.3 カテゴリ別領域

ピン留め領域の下に、用途別カテゴリを縦方向に表示する。

カテゴリ例：

- 業務
- 医療機器
- 電子カルテ
- AI
- Google
- 開発
- 自動化
- MBA
- 情報収集
- 個人
- 一時保存

カテゴリ数は必要に応じて追加する。

---

## 6. カード仕様

各URLはカードとして表示する。

### 表示項目

- ファビコン
- 表示名
- ドメイン名
- 任意の短い説明
- ピン留め状態

### 操作

- カード全体をクリック可能にする
- リンク先は新しいタブで開く
- マウスを重ねた際にカードを軽く強調する
- スマートフォンではタップしやすいサイズを確保する

### 表示しない項目

- Webページのリアルタイムプレビュー
- iframe
- 大きなサムネイル画像
- アクセス回数
- 複雑な利用履歴
- コメント機能
- ユーザー管理機能

「リンク先が何となく分かる」状態は、ファビコン、表示名、ドメイン名で実現する。

---

## 7. Googleスプレッドシート設計

### 管理列

| 列名 | 内容 | 入力例 |
|---|---|---|
| active | ページに表示するか | TRUE |
| pin | 最上部にピン留めするか | TRUE |
| category | 用途別カテゴリ | 開発 |
| name | カードに表示する名前 | GitHub |
| url | リンク先 | https://github.com/ |
| description | 任意の短い説明 | コード・リポジトリ管理 |
| order | 表示順 | 10 |

### 任意追加列

将来的に必要となった場合のみ追加する。

| 列名 | 内容 |
|---|---|
| created_at | 登録日 |
| updated_at | 更新日 |
| expires_at | 一時リンクの期限 |
| icon_url | 独自アイコン |
| note | 管理用メモ |

初期段階では追加しない。

---

## 8. URLの運用ルール

### 通常URLを登録する場合

```text
active = TRUE
pin = FALSE
category = 該当カテゴリ
```

### 一時URLを登録する場合

```text
active = TRUE
pin = TRUE
category = 一時保存
```

### タスク完了後も残す場合

```text
active = TRUE
pin = FALSE
category = 正式なカテゴリへ変更
```

### タスク完了後に使用しない場合

```text
active = FALSE
pin = FALSE
```

行自体は削除せず、非表示にして履歴を残す。

不要なデータが増えた場合のみ、定期的に整理する。

---

## 9. デザイン方針

### 基本デザイン

- カード型
- 余白を広めに取る
- 文字情報を最小限にする
- カテゴリ見出しを明確にする
- ピン留め領域を視覚的に区別する
- 端末の画面幅に応じてカード数を自動調整する

### PC表示

- 1行に3～5枚程度
- 横幅220～280px程度
- 縦方向にカテゴリを並べる

### スマートフォン表示

- 1行に1～2枚
- 画面幅に応じて自動調整
- 横スクロールは使用しない

### ダークモード

初期段階から端末設定に自動追従させる。

手動切り替えボタンは設置しない。

---

## 10. 負荷軽減方針

- JavaScriptライブラリは原則使用しない
- HTML、CSS、標準JavaScriptのみで作成する
- データ取得はページ表示時に1回だけ行う
- 画像はファビコンのみ使用する
- 不要なアニメーションを使用しない
- スプレッドシートのデータ量を必要最小限にする
- キャッシュを利用できる設計にする

---

## 11. セキュリティ方針

### 登録してよい情報

- 一般公開サービスのURL
- ログイン画面へのURL
- 公開Webページ
- 個人利用のクラウドサービス入口
- 機密情報を含まないGoogle Driveリンク

### 登録しない情報

- パスワード
- APIキー
- アクセストークン
- n8nのWebhook URL
- 認証情報入りURL
- 患者情報を含むURL
- 個人情報がURLパラメータに含まれるリンク
- 院内限定システムの管理画面URL
- サーバー管理画面
- ローカルIPアドレス
- 機密性の高いGoogle Driveファイル
- Zoomのパスコード付き参加URL

認証情報はブラウザまたはパスワードマネージャーで管理し、リンク集には保存しない。

---

## 12. 検索エンジン対策

完全な非公開化はできないが、検索結果に表示されにくくする。

実施内容：

- `robots.txt` を配置する
- HTMLに `noindex` と `nofollow` を設定する
- GitHubリポジトリへの説明文を最小限にする
- 公開サイトからリンクしない
- サイトマップを作成しない

これらは検索エンジンへの依頼であり、アクセス制御ではない。

### フェーズ1：最小構成

以下を実装する。

- Googleスプレッドシート作成
- GASによるJSON出力
- GitHub Pages作成
- カード型表示
- カテゴリ別表示

候補：

- 最終更新日時

- ピン留めの表示順調整

- ブックマークレット

- GitHub Actionsによる静的JSON生成

---

## 15. 完了条件

以下を満たした時点で初期構築完了とする。

- GoogleスプレッドシートからURLを追加できる
- スプレッドシートの変更がWebページへ反映される
- ピン留めURLが最上部に表示される
- 通常URLがカテゴリ別に表示される
- カード全体をクリックして新規タブで開ける
- PCとスマートフォンで崩れずに表示される
- URL以外の機密情報を保持しない
- 検索エンジン向けのインデックス抑制を設定する
- 表示に不要な機能やライブラリを使用しない

---

## 16. 初期方針の結論

本プロジェクトは、Googleスプレッドシートを管理画面、GitHub Pagesを表示画面として構築する。

初期版では機能を増やさず、以下の4機能に限定する。

1. カテゴリ別表示
2. カード型リンク
3. ピン留め
4. 表示・非表示管理

検索、ログイン、利用履歴、リアルタイムプレビューなどは実装しない。

実際の利用を開始した後、運用上の不便が確認された場合のみ追加開発を行う。
